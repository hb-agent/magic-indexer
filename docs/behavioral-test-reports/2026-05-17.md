**E6 — Keyset cursor is deterministic.**

Two-page test:
- Page 1: 5 URIs, last is `at://...3mm2hbpx3dk2e`, `endCursor` is
  a base64 of `["indexed_at","2026-05-17T13:20:31Z","at://.../3mm2hbpx3dk2e"]`.

`(indexed_at, uri)` tuple it encodes.

## Catalogue amendments

These changes are recommended for `docs/behavioral-tests.md` based
on what this first run encountered. They're noted here rather
than applied directly so the catalogue change is a deliberate,
reviewed edit.

1. **E1 — fix the example query.** The catalogue uses `indexedAt`
   on `GenericRecord` which doesn't exist. Replace with `uri did
   rkey` (the always-present scalar fields). Mention in the test
   that `value` is exposed via the per-collection auto-generated
   types (e.g. `orgHypercertsClaimActivity { value }`) but not on
   the polymorphic `records` query.

2. **E2 / E3 — use the per-collection resolver.** The generic
   `records` query does not accept `authors`. Filter args live on
   the auto-generated per-collection resolvers
   (`orgHypercertsClaimActivity(authors: [...])`). Update both
   tests to use that shape, and add a sentence about why (record
   filtering is per-collection because the field schemas differ).

3. **A3 — `CounterVec` series may be absent until first use.**
   Add a note: "`hypergoat_ingestion_error_total{consumer=...}` is
   a labelled CounterVec — it does not appear in `/metrics`
   output until the first error per consumer label. Treat its
   absence as 'no errors yet,' not as a missing metric. To
   force-initialise the series for the test, an alternative is
   to trigger a controlled error (e.g. via the C4 LOCAL test)."

4. **E5 — fix the example query.** `value` is on per-collection
   types but not always present. Replace with the always-present
   `uri did` so the test compiles regardless of which collection
   is queried.

5. **E7 — field name is `contributor` (singular).** The catalogue
   shows `contributors: {eq: ...}` but the actual field is
   `contributor: {eq: ...}`. This mismatch isn't a contract
   change — the field has always been singular in the
   `OrgHypercertsClaimActivityWhereInput`; the catalogue text was
   guessing.

6. **C1 — assertion should be metric-based, not stats-based.**
   `/stats records` lags behind ingestion when events are
   updates-only or create+delete pairs. Strengthen the expected
   result to: "`hypergoat_jetstream_events_total` increased by ≥1
   for at least one collection." Keep `/stats records` as a soft
   signal, not a hard assertion.

7. **General — provide a `LOCAL` setup runbook.** ~50% of the
   catalogue is skipped here because the dev container has no
   Docker. Either point to an existing setup doc explicitly in
   the "How to run" section, or add a one-page "Local stack for
   behavioral tests" doc and link to it. The current
   `docker compose -f docker-compose.postgres.yml up -d` line
   doesn't help in an environment without Docker installed.
